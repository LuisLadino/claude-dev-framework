## Framework vs Project Files

This `.claude/` folder comes from the framework repo. Some files are shared (framework-owned), others are project-specific.

**Framework-owned (synced from framework repo):**
- `CLAUDE.md` - these instructions
- `commands/` - utility commands (analyze, audit, learn, init-project, sync-stack)
- `hooks/` - all hooks
- `skills/` - all skills
- `agents/` - agent definitions

**Project-specific (NEVER overwrite when syncing):**
- `docs/` - project documentation and research
- `specs/` - project rules and patterns, generated by /sync-stack
- `session-state.json` - runtime state
- `settings.local.json` - per-project permissions
- `research/` - project research

**To update framework in a project:**
1. Copy only framework-owned files from the framework repo
2. Do NOT copy specs/ - run `/sync-stack` instead to regenerate for this project
3. Preserve any project-specific customizations

## Workflow

**Setup (once per project):**
Discuss → Document → Create repo → Copy framework → /init-project → /sync-stack → Set up GitHub with /plan

**Working (repeatable):**
GitHub Issue → /research → /define → /ideate → /build → /test → /commit → Merge

Skills map to design thinking phases. /research is the entry point (understand the problem). /build is the commitment point (create branch, start making). Issue moves to "In Progress" at /build. Commit skill handles push + PR. Issue auto-closes on merge.

**Planning (anytime):**
Use /plan to create issues, review backlog, prioritize, manage milestones. GitHub Issues are the system of record.

## GitHub Issues as Design Thinking Records

IMPORTANT: GitHub issues are not just task trackers. They are the qualitative record of how we think through problems. Every issue should capture the design thinking journey: the research, the reasoning, the alternatives considered, and why we made the decisions we made.

**When writing or updating issues:**
- Document the WHY, not just the WHAT. "Updated auth middleware" is a changelog entry. "Chose JWT over session tokens because of stateless scaling requirements" is a decision record.
- Include alternatives considered and why they were rejected
- Capture research findings and evidence that informed the decision
- Write discoveries and reasoning as they happen, not just the final answer
- Use the design thinking phases to structure the journey

**When to create a new issue vs continue the current one:**

STAY on the current issue when:
- Same root problem we're solving
- Issue found while testing our solution
- Refinement of approach: first attempt didn't work
- Definition of Done not yet met

CREATE a new issue when:
- Different Definition of Done than the current issue
- Could be solved independently of current work
- Different component or domain affected
- Would significantly expand current scope

<example>
Good issue update:
"Investigated three approaches for context injection:
1. Hook reads file and injects content: simplest, but no evaluation
2. Agent spawns at session start to evaluate: rich context, but 30-45s latency
3. Hybrid: hook injects raw content, agent evaluates in background

Going with option 3 because it gives immediate context (no blank start) while the agent adds strategic evaluation asynchronously. Trade-off: agent results arrive after first response."
</example>

**When to close an issue:**

IMPORTANT: NEVER close an issue until the fix has been verified and tested. Writing code that should fix something is not the same as confirming it works. Many fixes require a new session to test (e.g., hook changes, session start behavior, context injection). If you cannot verify the fix in the current session, leave the issue open and document what was done and what still needs testing.

Do NOT use "Closes #X" in commits unless the fix has been tested and confirmed working.

## Specs

Specs define project rules and patterns. They live in `.claude/specs/`. Specs are not limited to code. They can cover any convention, process, or standard for the project.

Before making changes, read the relevant specs. The enforce-specs hook will block edits until you do.

To generate specs: `/sync-stack`
To add a library: `/sync-stack prisma`
To add custom rules: `/sync-stack --custom api-conventions`

---

## Hooks

Hooks enforce behavior. Don't fight them.

- **enforce-specs** blocks edits until specs are read
- **enforce-skills** blocks git commit, requires Skill tool invocation for full workflow
- **enforce-plan** blocks `gh issue create` until plan skill is read (ensures issues have proper context)
- **block-dangerous** blocks rm -rf, force push, credential exposure
- **verify-before-stop** checks for debug statements before stopping
- **awareness** prompts for /analyze when issues accumulate

If a hook blocks you, there's a reason.

## Writing for Luis

When writing content that other people will read on Luis's behalf (articles, emails, posts, bios, applications, portfolio content), follow these rules:

**Voice:**
- Direct and clear. Plain language. No clever phrasing.
- Honest and grounded. No superlatives without evidence.
- Evidence-based. Specific examples, not adjectives. Show, don't tell.
- Warm but professional. Not casual, not stiff.

**Luis says:** "I don't know" (when true), "I figured it out", "I learned...", "The constraint was...", "What worked was...", "The trade-off was..." Short sentences. Active voice. Contractions.

**Luis NEVER says:** "I'm passionate about...", "leverage/synergize/optimize", "world-class/best-in-class", "ninja/rockstar/guru", "utilize" (say "use"), "ensure" (say "make sure").

**Formatting:**
- No em dashes. Use periods or colons instead.
- Varied sentence length. Mix short and medium.
- No scaffolding phrases ("Here's what I found:", "Let me explain:")
- No filler praise ("solid", "comprehensive", "well-structured")

**Technical vocabulary:** Use domain-appropriate terms when they're the right word for the audience. "Heuristic evaluation" is precise. "Leveraging synergies" is noise.

**Quality check before finalizing:** Sounds like a human? No em dashes? No corporate speak? Active voice? Specific examples? Varied sentence length? Would Luis actually say this?
